Current Manual Process

ICE sends daily spreadsheet containing patient encounter data. PRM imports this into the Electronic Billing System (EBS), which applies business rules to create billable claims.

Manual reconciliation required: ICE manually prepares a Daily Reconciliation spreadsheet, PRM manually appends billing counts, and daily 3 PM meetings review discrepancies.

üí° Proposed Solution

üöÄ
Automated Reconciliation System
The solution will automatically process ICE export files, evaluate billing success/failure for each encounter, generate structured output files, and maintain a persistent historical ledger of unbillable encounters.

üîÑ Automated Workflow

üì§
Step 1: ICE Sends File
ICE sends daily spreadsheet: Export PRM mm-dd-yyyy Completed.xlsx containing encounter data for multiple service dates.
‚öôÔ∏è
Step 2: Automated Processing
System automatically processes the file, validates data, and imports into EBS for billing evaluation.
üí≥
Step 3: Billing Evaluation
EBS evaluates each encounter and determines if it can be billed. System captures success/failure status and reasons.
üìä
Step 4: Generate Output Files
System generates two files: General Reconciliation (daily summary) and Master Missing (historical ledger).
üìß
Step 5: Automated Delivery
Files are automatically delivered to ICE via email and/or SFTP for review.

üìÑ Output Files

üìã
General Reconciliation File

File Name: General Reconciliation mm-dd-yyyy.xlsx

Purpose: Provides ICE with a complete, automated summary of every encounter sent, whether it was billed, and why certain encounters could not be billed.

Contains:

  • Data Sheet: All original fields + Billed status + Reason for not billed
  • Summary Sheet: Aggregated daily metrics by date, facility, provider, and type of care
üìö
Master Missing File

File Name: Master Missing to mm-dd-yyyy.xlsx

Purpose: Serves as the historical ledger of all encounters that could not be billed due to missing or invalid data, updated cumulatively across all days.

Behavior:

  • Persistently updated each time the import runs
  • If encounter remains incomplete, record is updated with new attempt date
  • If encounter becomes billable, it is removed from Master Missing
File Sheet Key Fields Purpose
General Reconciliation Data All original fields + Billed (Yes/No) + Reason for not billed Complete line-by-line reconciliation
General Reconciliation Summary Date, Facility, Provider, Type of Care, PRM Billing Count, CPTs Aggregated daily metrics
Master Missing Data Patient Name, DOB, Date of Service, Last Attempt Date, Billed (No), Reason Historical ledger of incomplete encounters

üìê Business Rules

üîç
Encounter Evaluation
Each encounter is evaluated by EBS to determine if it can be billed
‚úÖ
Billing Success
If all required data present ‚Üí Marked as "Billed = Yes"
‚ùå
Billing Failure
If data missing ‚Üí Marked as "Billed = No" with reason
üìù
Reason Capture
System captures specific reason (e.g., "Missing DX", "Missing CPT")
Condition Billing Result Reason Code Action
All required data present Billed = Yes N/A Claim created in EBS
Missing DX (Diagnosis) codes Billed = No "Missing DX" Added to Master Missing
Missing CPT codes Billed = No "Missing CPT" Added to Master Missing
Provider mismatch Billed = No "Provider validation failed" Added to Master Missing
Invalid facility Billed = No "Invalid Facility" Added to Master Missing

üîÑ Master Missing File Behavior

üì•
New Encounter Cannot Be Billed
When an encounter fails billing evaluation, it is added to the Master Missing file with the current date as "Last Attempt to Process" and the specific reason for failure.
üîÑ
Encounter Reappears in Later File
If the same encounter (identified by unique key) appears in a future import file, the system updates the existing Master Missing record with the new "Last Attempt to Process" date and updated reason.
‚úÖ
Encounter Becomes Billable
If an encounter that was previously in Master Missing now successfully bills, it is automatically removed from the Master Missing file and appears in the General Reconciliation file with "Billed = Yes".
üí°
Key Benefit
This mechanism ensures no encounter is ever lost, even if ICE resends incomplete records weeks or months later. The Master Missing file serves as a complete historical audit trail of all unbillable encounters.
