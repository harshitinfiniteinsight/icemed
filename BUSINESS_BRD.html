Executive Summary

Project Purpose
This project automates the ICE Charge Import and Reconciliation Process, eliminating approximately 3 hours of daily manual work while providing ICE with structured, automated reconciliation outputs and maintaining a comprehensive historical ledger of unbillable encounters.

Eliminate 3 hours/day of manual reconciliation work

Current Situation

Current Manual Process

ICE sends daily spreadsheet containing patient encounter data. PRM imports this into the Electronic Billing System (EBS), which applies business rules to create billable claims.

Manual reconciliation required: ICE manually prepares a Daily Reconciliation spreadsheet, PRM manually appends billing counts, and daily 3 PM meetings review discrepancies.

Time Required: 3 hours per day

Problem Area Description Business Impact
Manual Reconciliation Time-consuming, error-prone, not scalable 3 hours/day lost productivity
Missing Information Tracking No centralized historical record of incomplete encounters Encounters lost, no audit trail
Lack of Automated Feedback ICE cannot easily reconcile what was billed vs. received vs. missing Delayed issue resolution
Operational Dependency Daily reconciliations consume significant staff time Bottleneck in workflow
Multi-Date Files Input files mix encounters from multiple service dates Complex tracking requirements

Proposed Solution

Automated Reconciliation System
The solution will automatically process ICE export files, evaluate billing success/failure for each encounter, generate structured output files, and maintain a persistent historical ledger of unbillable encounters.

Automated Workflow

Step 1: ICE Sends File
ICE sends daily spreadsheet: Export PRM mm-dd-yyyy Completed.xlsx containing encounter data for multiple service dates. File is delivered via secure channel (email or SFTP).
Step 2: Automated Processing
System automatically processes the file, validates data structure and content, parses all encounter records, and prepares data for EBS import.
Step 3: Billing Evaluation
EBS evaluates each encounter and determines if it can be billed. System captures success/failure status and specific reasons for failures (e.g., "Missing DX", "Missing CPT").
Step 4: Generate Output Files
System generates two files: General Reconciliation (daily summary with all encounters and billing status) and Master Missing (cumulative historical ledger of unbillable encounters).
Step 5: Automated Delivery
Files are automatically delivered to ICE via email and/or SFTP. Delivery confirmation is logged and tracked. ICE receives notification of file availability.

User Flows & Scenarios

Scenario 1: Successful Processing (Happy Path)
Success
1
ICE Uploads File
ICE operations team uploads Export PRM 12-10-2025 Completed.xlsx containing 150 encounters with complete data (DX codes, CPT codes, provider information).
2
System Validates File
System validates file format, checks for required columns, validates data types. All validations pass. File is accepted for processing.
3
EBS Billing Evaluation
System sends each encounter to EBS for billing evaluation. EBS processes all 150 encounters successfully. All encounters receive "Billed = Yes" status.
4
Generate General Reconciliation
System generates General Reconciliation file with Data sheet (150 rows, all marked "Billed = Yes") and Summary sheet (aggregated metrics by date/facility/provider).

6
Deliver Files to ICE
System automatically emails General Reconciliation file to ICE operations team. File is also uploaded to SFTP server. ICE receives notification.
Scenario 2: Partial Success (Some Encounters Fail)
Warning

ICE Uploads File
ICE uploads file with 200 encounters. 150 encounters have complete data, 50 encounters are missing DX codes or CPT codes.
2
System Validates File
File structure is valid. System accepts file but flags 50 encounters with missing data during validation phase.
3

150 encounters bill successfully ("Billed = Yes"). 50 encounters fail billing evaluation with reasons: 30 "Missing DX", 20 "Missing CPT".
4
Generate General Reconciliation
General Reconciliation file shows 150 "Billed = Yes" and 50 "Billed = No" with specific reasons. Summary sheet reflects partial billing success.
5
Update Master Missing
System checks if 50 failed encounters exist in Master Missing. 20 are new (added with current date), 30 are existing (updated with new attempt date and reason).
6
Deliver Files to ICE
Both General Reconciliation and updated Master Missing files are delivered. ICE can see which encounters failed and why, enabling them to correct and resubmit.
Scenario 3: Previously Failed Encounter Now Succeeds
Success
1
Previous State
Encounter "Patient A, DOB 01/15/1950, DOS 11/17/2025, Facility: Azure Shores" was in Master Missing with reason "Missing DX" from previous import on 12/01/2025.
2
ICE Corrects and Resubmits
ICE adds missing DX code to the encounter and includes it in new file Export PRM 12-10-2025 Completed.xlsx.
3
System Processes File
System generates encounter key and matches it to existing Master Missing entry. Encounter is sent to EBS for evaluation.
4
EBS Billing Success
EBS evaluates encounter with complete data and returns "Billed = Yes". Claim is successfully created.
5
Remove from Master Missing
System automatically removes this encounter from Master Missing file since it is now billable. Updated Master Missing file reflects removal.
6
General Reconciliation Shows Success
General Reconciliation file shows this encounter with "Billed = Yes". ICE can see that the previously failed encounter is now successfully processed.
Scenario 4: File Validation Failure
Error
1
ICE Uploads Invalid File
ICE uploads file with missing required columns (e.g., "Patient Name" or "Date of Service" columns are missing).
2
System Validates File
System performs validation checks and identifies missing required columns. Validation fails.
3
Error Notification
System generates error report listing all validation failures (missing columns, invalid data types, etc.). Processing stops.
4
Notify ICE
System sends error notification to ICE operations team with detailed error report. ICE can correct file and resubmit.
5
No Files Generated
No General Reconciliation or Master Missing files are generated. System logs error for monitoring and audit purposes.

Output Files

General Reconciliation File

File Name: General Reconciliation mm-dd-yyyy.xlsx

Purpose: Provides ICE with a complete, automated summary of every encounter sent, whether it was billed, and why certain encounters could not be billed.

Contains:

  • Data Sheet: All original fields + Billed status + Reason for not billed
  • Summary Sheet: Aggregated daily metrics by date, facility, provider, and type of care
Master Missing File

File Name: Master Missing to mm-dd-yyyy.xlsx

Purpose: Serves as the historical ledger of all encounters that could not be billed due to missing or invalid data, updated cumulatively across all days.

Behavior:

  • Persistently updated each time the import runs
  • If encounter remains incomplete, record is updated with new attempt date
  • If encounter becomes billable, it is removed from Master Missing
File Sheet Key Fields Purpose
General Reconciliation Data All original fields + Billed (Yes/No) + Reason for not billed Complete line-by-line reconciliation
General Reconciliation Summary Date, Facility, Provider, Type of Care, PRM Billing Count, CPTs Aggregated daily metrics
Master Missing Data Patient Name, DOB, Date of Service, Last Attempt Date, Billed (No), Reason Historical ledger of incomplete encounters

Master Missing File Behavior

New Encounter Cannot Be Billed
When an encounter fails billing evaluation, it is added to the Master Missing file with the current date as "Last Attempt to Process" and the specific reason for failure.
Encounter Reappears in Later File
If the same encounter (identified by unique key) appears in a future import file, the system updates the existing Master Missing record with the new "Last Attempt to Process" date and updated reason.
Encounter Becomes Billable
If an encounter that was previously in Master Missing now successfully bills, it is automatically removed from the Master Missing file and appears in the General Reconciliation file with "Billed = Yes".
Key Benefit
This mechanism ensures no encounter is ever lost, even if ICE resends incomplete records weeks or months later. The Master Missing file serves as a complete historical audit trail of all unbillable encounters.

Scope of Work

In Scope
  • Intake and processing of ICE export Excel files
  • Automated generation of General Reconciliation file
  • Automated generation of Master Missing file
  • Integration with existing Charge Import Tool
  • EBS billing evaluation integration
  • Automated file delivery to ICE
  • Error handling and validation
  • Logging and monitoring
Out of Scope
  • Modifications to ICE's internal systems
  • Changes to EBS beyond data import logic
  • Workflows unrelated to billing readiness
  • Real-time processing (batch only)
  • Mobile application
  • Direct database access by ICE

Risks & Mitigations

Risk Impact Mitigation Strategy
Inconsistent data format from ICE High Add input validation + error logging with clear error messages
Failure to maintain Master Missing history High Implement database/file-level versioning and backup procedures
Business rule changes in EBS Medium Publish change protocol with ICE and maintain flexible integration
EBS system downtime Medium Implement retry logic and queue failed encounters for reprocessing
